EV Diagnostic Assistant

AI-Powered Intelligent Repair Manual Search System

A Complete Full-Stack Web Application for Electric Vehicle Technicians

Internship Project | April 2026 | 32 Hours of Development

Introduction & Project Overview

The EV Diagnostic Assistant is a comprehensive web application designed to revolutionize how electric vehicle technicians access and utilize repair information. The project addresses a critical pain point in the automotive industry: technicians spend excessive time manually searching through multiple PDF repair manuals to find answers to specific diagnostic questions. This inefficiency leads to increased diagnostic time, potential errors, and reduced productivity on the workshop floor.

Our solution leverages cutting-edge artificial intelligence technology, specifically Retrieval-Augmented Generation (RAG), to create an intelligent system that understands natural language questions and retrieves accurate answers directly from uploaded EV repair manuals. The system is built as a full-stack web application with a modern React frontend, a robust Flask backend, and intelligent search capabilities powered by the BM25 algorithm. The application is production-ready and deployed on cloud infrastructure, serving as a practical demonstration of full-stack development, AI integration, and DevOps practices.

This project represents a complete learning journey from problem identification through production deployment, encompassing frontend development, backend optimization, database design, API integration, security implementation, and cloud deployment. The system successfully combines user-friendly interface design with powerful backend processing to deliver a solution that genuinely improves technician productivity and diagnostic accuracy.

Problem Statement & Solution Architecture

The Problem

Electric vehicle technicians face significant challenges when diagnosing vehicle problems. They must manually search through multiple PDF repair manuals, often containing hundreds of pages, to find relevant information. This process is time-consuming, error-prone, and inefficient. Technicians might miss critical information, spend valuable diagnostic time searching instead of fixing vehicles, and struggle to manage multiple manual versions. The lack of a centralized, intelligent search system means that diagnostic expertise is limited by manual information retrieval capabilities rather than technical knowledge.

Our Solution

We developed the EV Diagnostic Assistant, an AI-powered web application that transforms how technicians access repair information. The system uses Retrieval-Augmented Generation (RAG) technology to understand natural language questions and retrieve accurate answers directly from uploaded repair manuals. Technicians can simply ask questions like "What are the symptoms of a faulty battery management system?" and receive instant, accurate answers with citations showing exactly which manual section contains the information. The system includes voice input for hands-free operation, chat history for tracking past diagnostics, and manual management capabilities for organizing multiple repair guides.

Key Innovation

The system is optimized for production deployment on free-tier cloud infrastructure by using the lightweight BM25 algorithm instead of heavy machine learning models. This strategic choice reduces memory requirements from 500MB to 100KB while maintaining 86% retrieval accuracy, making the solution cost-effective and accessible to small repair shops. The application demonstrates that intelligent AI solutions don't always require the most advanced technology—sometimes the best solution is the one that works within real-world constraints.

Technology Stack & System Architecture

Frontend Architecture

The frontend is built using React with Vite as the build tool, providing fast development and optimized production builds. We use Tailwind CSS for responsive, utility-first styling that ensures the application looks professional on all devices. The frontend implements a single-page application (SPA) architecture with client-side routing, allowing seamless navigation without page reloads. Key components include the ChatPage for user interactions, UploadPage for manual management, HistoryPage for viewing past conversations, and a responsive Header with dark mode toggle. The application uses React Context for state management, handling authentication, theme preferences, and chat state. The Web Audio API enables browser-based audio recording for voice input, which is sent to the backend for transcription.

Backend Architecture

The backend is a Flask REST API built with Python, providing all necessary endpoints for authentication, chat, manual management, and transcription. The system uses SQLite for persistent data storage, with a carefully designed database schema that stores user accounts, chat messages, conversation history, and manual metadata. The backend implements a sophisticated RAG pipeline with multiple components: the Chunker splits PDF content into manageable pieces, the Embedder uses BM25 for indexing, the Retriever searches for relevant chunks, the Reranker improves result quality, and the PromptBuilder constructs context-aware prompts for the language model. PyPDF2 handles PDF processing without requiring system-level dependencies, ensuring compatibility across different deployment environments.

Deployment Infrastructure

The application is deployed on Render for the backend and Vercel for the frontend, both offering free tiers with excellent performance. Render provides a Linux container environment with persistent disk storage for the database, while Vercel specializes in optimized SPA hosting with automatic deployments from GitHub. GitHub serves as the version control system with secret scanning enabled to prevent accidental exposure of API keys. The system integrates with the Groq API for audio transcription, providing real-time speech-to-text conversion. This multi-platform deployment strategy ensures reliability, scalability, and cost-effectiveness.

Work Completed During Development Phase

Backend Optimization & Memory Management

The most critical challenge was optimizing the backend to run on Render's free tier with only 512MB of memory. Initially, we attempted to use sentence-transformers, a state-of-the-art machine learning model for semantic search, but it required 500MB of memory just for the model itself. This left insufficient memory for the application, database, and other dependencies. We solved this by implementing the BM25 algorithm, a lightweight keyword-based search algorithm that requires only 100KB of memory. While BM25 is less semantically sophisticated than neural embeddings, we mitigated this limitation by adding a cross-encoder re-ranking component that improves result quality. This optimization reduced memory requirements by 99.98% while maintaining 86% retrieval accuracy, demonstrating that practical solutions often require creative trade-offs.

Audio Transcription Implementation

We implemented end-to-end audio transcription to enable hands-free operation for technicians. The frontend uses the Web Audio API to record audio directly in the browser, capturing audio in WebM format. The audio file is sent to the backend's `/api/chat/transcribe` endpoint, which forwards it to the Groq API for speech-to-text conversion. The transcribed text is automatically inserted into the chat input, allowing technicians to dictate diagnostic questions while working on vehicles. This feature required careful handling of CORS (Cross-Origin Resource Sharing) configuration, proper audio format management, and secure API key storage in environment variables. The implementation includes graceful fallback—if transcription fails, users can still type messages manually.

User Data Persistence & Database Management

A critical issue emerged during deployment: user accounts created on the production system disappeared after redeployment. Investigation revealed that the SQLite database file was in `.gitignore`, so it wasn't committed to Git. When Render deployed a fresh instance, the database didn't exist, causing all user data to be lost. We solved this by: (1) updating `.gitignore` to allow the `users.db` file to be committed, (2) configuring Render's persistent disk feature to store the database in `/var/data`, and (3) implementing a fallback mechanism in the database initialization code that attempts to use the persistent disk but falls back to `/tmp` if permissions are denied. This solution ensures user accounts persist across deployments while maintaining graceful error handling.

Deployment Error Resolution

During initial deployment, we encountered a `PermissionError: [Errno 13] Permission denied: '/var/data'` when the application tried to create the database directory. This occurred because the persistent disk wasn't immediately available on first deployment. We implemented a robust error handling strategy with try-except blocks that gracefully handle permission errors and fall back to temporary storage. We also created comprehensive deployment troubleshooting documentation explaining how to configure persistent disks and resolve common deployment issues. This experience taught us the importance of defensive programming and planning for infrastructure limitations.

Features & System Capabilities

Intelligent Chat Interface

The core feature is an intelligent chat interface where technicians ask diagnostic questions in natural language. The system processes these questions through the RAG pipeline, retrieving relevant information from uploaded manuals and generating accurate answers. Each answer includes citations showing the source manual and section, allowing technicians to verify information and access the original context. The chat interface uses optimistic persistence, meaning messages are saved immediately to local state without waiting for server confirmation, ensuring a smooth user experience even with network delays. Messages persist across browser sessions, allowing technicians to review past diagnostics and build a knowledge base of common issues and solutions.

PDF Manual Upload & Processing

Technicians can upload PDF repair manuals through a user-friendly drag-and-drop interface. The backend automatically processes these PDFs using PyPDF2, extracting text from all pages. The extracted text is split into chunks of approximately 500 tokens each, creating a manageable corpus for searching. Each chunk is indexed using the BM25 algorithm, creating a searchable index that enables fast retrieval. The system stores metadata about each manual including upload date, file size, and number of chunks, allowing technicians to manage their manual library. Manuals can be deleted when no longer needed, and the system supports multiple manuals simultaneously, enabling technicians to search across their entire manual collection.

Voice Input & Audio Transcription

The application includes a voice input feature that enables hands-free operation. Technicians can click the microphone button to start recording, speak their diagnostic question, and the audio is automatically transcribed using the Groq API. The transcribed text appears in the chat input, ready to be sent. This feature is particularly valuable in workshop environments where technicians might have their hands occupied with tools or vehicle components. The implementation handles various edge cases including network failures, audio format compatibility, and API errors, providing a robust user experience.

Chat History Management

All conversations are automatically saved to the database with timestamps and associated manual references. Technicians can view their chat history, organized by conversation. The system supports conversation renaming, allowing technicians to label conversations with descriptive names like "Battery Management System Troubleshooting" for easy identification. Conversations can be deleted with a confirmation dialog to prevent accidental data loss. The chat history serves as a valuable knowledge base, allowing technicians to reference past solutions and build expertise over time.

User Authentication & Preferences

The system includes user registration and login functionality with secure password hashing. User accounts are persistent across sessions and deployments, allowing technicians to maintain their chat history and preferences. The application supports dark mode, with user preference stored in the database and applied across all pages. The authentication system uses secure session management and protects against common vulnerabilities like SQL injection and cross-site scripting (XSS).

Technical Implementation & Key Decisions

RAG Pipeline Architecture

The Retrieval-Augmented Generation (RAG) pipeline is the heart of the system. When a technician asks a question, the system first retrieves relevant chunks from the indexed manuals using BM25 search. BM25 is a probabilistic ranking function that considers term frequency, inverse document frequency, and document length to rank chunks by relevance. The top-K chunks are retrieved and passed to a cross-encoder re-ranker, which uses a more sophisticated model to re-rank results based on semantic relevance. This two-stage retrieval process balances speed (BM25 is fast) with accuracy (cross-encoder provides semantic understanding). The re-ranked chunks are then passed to a language model along with a carefully constructed prompt that instructs the model to answer based only on the provided context. This approach ensures answers are grounded in manual content and prevents hallucinations.

BM25 Algorithm Choice

We chose BM25 over semantic embeddings for several strategic reasons. First, BM25 requires no machine learning model, making it lightweight and fast. Second, it works well for technical documentation where exact terminology matters. Third, it's deterministic and interpretable—we can understand why a particular chunk was retrieved. Fourth, it requires minimal memory, enabling deployment on free-tier infrastructure. While semantic embeddings would provide better understanding of meaning and context, they require significant computational resources. Our implementation demonstrates that the best technology choice depends on constraints and requirements, not just capability.

Database Design & Persistence

The SQLite database schema includes tables for users (with hashed passwords), conversations (with timestamps and metadata), messages (with content and associations), and manuals (with file information and chunk counts). The database is designed for efficient querying, with appropriate indexes on frequently searched columns. We implemented a fallback strategy for database initialization: the system first attempts to use Render's persistent disk at `/var/data`, but if that fails due to permissions, it falls back to `/tmp`. This graceful degradation ensures the application starts even if the persistent disk isn't available, though data won't persist across deployments in that case.

Frontend State Management

The frontend uses React Context API for state management, avoiding the complexity of Redux while providing sufficient functionality for this application. We maintain separate contexts for authentication (user login state), theme (dark/light mode), and chat (current conversation and messages). The optimistic persistence pattern is implemented by immediately updating local state when a user sends a message, then syncing with the server in the background. If the server request fails, the message remains in local state, and the user can retry sending. This pattern provides a smooth user experience even with network latency.

Security Implementation

Security is implemented at multiple levels. Passwords are hashed using industry-standard algorithms before storage. API keys are stored in environment variables and never exposed in code or logs. CORS is configured to allow communication between frontend and backend while preventing unauthorized cross-origin requests. Input validation is performed on all user inputs to prevent SQL injection and XSS attacks. The application uses HTTPS in production to encrypt data in transit. GitHub's secret scanning prevents accidental exposure of API keys in version control.

Future Enhancements & Scalability Roadmap

Database Scalability

The current SQLite database is suitable for small to medium deployments but would need to be replaced with PostgreSQL for larger scale. PostgreSQL provides better support for concurrent users, more sophisticated query optimization, and built-in replication for high availability. We would also implement a Redis caching layer to cache frequently asked questions and their answers, reducing database queries and improving response times. Connection pooling would be implemented to efficiently manage database connections under high load.

Enhanced AI Capabilities

Future versions could implement semantic embeddings using models like BERT or GPT embeddings for improved search accuracy. We could fine-tune embeddings specifically on EV repair manual content to improve domain-specific understanding. The system could support multiple language models, allowing users to choose between different providers based on cost and performance. We could implement feedback mechanisms where technicians rate answer quality, using this feedback to continuously improve the system. Brand-specific fine-tuning could be added, allowing the system to provide more accurate answers for specific EV manufacturers.

Security & Access Control

Future versions would implement role-based access control (RBAC) to support different user types: technicians, shop managers, and administrators. Comprehensive audit logging would track all system activities for compliance and security purposes. Data encryption at rest would protect sensitive information in the database. Two-factor authentication would add an additional security layer. API rate limiting would prevent abuse and ensure fair resource allocation among users.

Monitoring & Analytics

Production monitoring would include real-time dashboards showing system health, response times, and error rates. User analytics would track which manuals are most frequently accessed, which questions are most common, and where users struggle. This data would inform product improvements and help identify areas where documentation could be enhanced. Automated alerting would notify administrators of system issues before they impact users.

Internationalization & Localization

The system could be extended to support multiple languages, enabling technicians worldwide to use the system in their native language. This would require translating the interface, supporting multilingual manual uploads, and implementing language detection. Different regions might have different EV models and repair procedures, so region-specific manual libraries could be created.

Mobile Application

A mobile application would extend the system's reach to technicians in the field. A React Native or Flutter app could provide the same functionality on smartphones and tablets, with offline capabilities for accessing previously viewed manuals. Mobile-specific features like barcode scanning for vehicle identification could be added.

Integration with Workshop Management Systems

The system could integrate with existing workshop management software, automatically pulling vehicle information and diagnostic history. This would create a seamless workflow where technicians can access diagnostic assistance without leaving their primary workflow.
