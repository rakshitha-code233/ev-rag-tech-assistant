# Manual Upload/Delete Bugfix Design

## Overview

This bugfix addresses two distinct but related issues in the manual management feature of the EV Diagnostic Assistant. Both bugs stem from incorrect HTTP header and URL encoding handling in the frontend service layer:

1. **Upload Bug**: The axios instance sets a default `Content-Type: application/json` header. When `manualService.js` attempts to override this with `Content-Type: multipart/form-data`, the browser-generated `boundary` parameter is stripped, causing Flask's multipart parser to fail.

2. **Delete Bug**: The `deleteManual` function applies `encodeURIComponent()` to the filename when constructing the DELETE URL. Flask's `<path:filename>` route converter receives the literal encoded string (e.g., `Tesla%20Model3.pdf`) instead of the decoded filename, causing a file-not-found error.

The fix strategy is minimal and surgical: remove the explicit `Content-Type` header from the upload request (allowing the browser to set it correctly with the boundary), and remove the `encodeURIComponent()` call from the delete URL (allowing axios to handle URL encoding at the transport level).

## Glossary

- **Bug_Condition (C)**: The condition that triggers either bug - uploading a PDF file OR deleting a manual with any filename
- **Property (P)**: The desired behavior - successful upload with correct multipart boundary OR successful delete with correct filename matching
- **Preservation**: All other API requests (login, register, chat, transcribe, list manuals, history) that must continue using `Content-Type: application/json`
- **multipart/form-data**: HTTP content type for file uploads that requires a `boundary` parameter to delimit form fields
- **boundary parameter**: A unique string generated by the browser to separate parts of a multipart request body (e.g., `----WebKitFormBoundary7MA4YWxkTrZu0gW`)
- **encodeURIComponent()**: JavaScript function that percent-encodes special characters in URL components
- **<path:filename>**: Flask route converter that captures the entire remaining URL path as a parameter, automatically URL-decoding it

## Bug Details

### Bug Condition

The bugs manifest in two distinct scenarios:

**Upload Bug**: When a user uploads a PDF file, the `uploadManual` function in `manualService.js` creates a FormData object and explicitly sets `Content-Type: multipart/form-data` in the request headers. However, the axios instance in `api.js` has a default `Content-Type: application/json` header. When axios merges these headers, the explicit override strips the browser-generated `boundary` parameter, resulting in a malformed multipart request that Flask cannot parse.

**Delete Bug**: When a user deletes a manual, the `deleteManual` function applies `encodeURIComponent(filename)` to construct the URL (e.g., `/api/manuals/Tesla%20Model3.pdf`). Axios then performs its own URL encoding at the transport level, but Flask's `<path:filename>` route converter receives the literal string `Tesla%20Model3.pdf` (with the percent-encoding preserved as literal characters) instead of the decoded `Tesla Model3.pdf`, causing the file lookup to fail.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { operation: string, file?: File, filename?: string }
  OUTPUT: boolean
  
  RETURN (input.operation == 'upload' AND input.file.type == 'application/pdf')
         OR (input.operation == 'delete' AND input.filename != null)
END FUNCTION
```

### Examples

**Upload Bug Examples:**
- User uploads `Tesla_Model3.pdf` → Backend receives multipart body without boundary → 400 Bad Request or parsing failure
- User uploads `ev_test_manual.pdf` → Backend cannot parse form data → Upload fails
- User uploads `Nissan Leaf.pdf` (with space) → Same multipart parsing failure → Upload fails

**Delete Bug Examples:**
- User deletes `Tesla Model3.pdf` → URL becomes `/api/manuals/Tesla%20Model3.pdf` → Flask looks for literal file `Tesla%20Model3.pdf` → 404 Not Found
- User deletes `ev_test_manual.pdf` → URL becomes `/api/manuals/ev_test_manual.pdf` → Works correctly (no special chars)
- User deletes `Owner's Manual.pdf` → URL becomes `/api/manuals/Owner%27s%20Manual.pdf` → Flask looks for `Owner%27s%20Manual.pdf` → 404 Not Found

**Edge Cases:**
- User uploads file with spaces in name → Should work after fix (multipart boundary present)
- User deletes file with only alphanumeric characters → Already works (no encoding needed)
- User deletes file with Unicode characters → Should work after fix (axios handles encoding)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- All other API requests (login, register, chat, transcribe, list manuals, history) must continue to send `Content-Type: application/json` and receive JSON responses
- The `listManuals` endpoint must continue to return an array of manual metadata without requiring multipart encoding
- All authenticated requests must continue to include the JWT token in the `Authorization: Bearer <token>` header
- The backend must continue to call `build_manual_index()` after successful upload or delete to update the RAG store

**Scope:**
All inputs that do NOT involve uploading a PDF file or deleting a manual should be completely unaffected by this fix. This includes:
- POST requests to `/api/auth/login` and `/api/auth/register`
- POST requests to `/api/chat` and `/api/chat/transcribe`
- GET requests to `/api/manuals`, `/api/history`, `/api/history/:id`
- POST requests to `/api/history`
- GET requests to `/api/health`

## Hypothesized Root Cause

Based on the bug description and code analysis, the root causes are confirmed:

1. **Upload Bug - Header Override Strips Boundary**: The axios instance in `api.js` sets a default `Content-Type: application/json` header. When `manualService.js` explicitly sets `Content-Type: multipart/form-data` in the upload request config, axios merges the headers but the explicit string value lacks the browser-generated boundary parameter. The browser only adds the boundary when it automatically sets the Content-Type header for FormData payloads. By explicitly overriding it, we prevent the browser from adding the boundary.

2. **Delete Bug - Double URL Encoding**: The `deleteManual` function calls `encodeURIComponent(filename)` to construct the URL path. Axios then performs its own URL encoding at the HTTP transport level (as all HTTP clients do). Flask's `<path:filename>` route converter receives the URL-decoded path segment, but because we pre-encoded it with `encodeURIComponent()`, Flask receives the literal percent-encoded string (e.g., `Tesla%20Model3.pdf`) instead of the decoded filename (e.g., `Tesla Model3.pdf`). The file system lookup fails because the actual file is named `Tesla Model3.pdf`, not `Tesla%20Model3.pdf`.

## Correctness Properties

Property 1: Bug Condition - Upload Success with Correct Boundary

_For any_ PDF file upload request, the fixed uploadManual function SHALL send the multipart/form-data request with the browser-generated boundary parameter intact, and the backend SHALL successfully parse the form data and save the file to the data/manuals/ directory.

**Validates: Requirements 2.1, 2.3**

Property 2: Bug Condition - Delete Success with Correct Filename

_For any_ manual delete request with a filename containing spaces or special characters, the fixed deleteManual function SHALL construct the URL without pre-encoding the filename, allowing axios to handle URL encoding at the transport level, and the backend SHALL receive the correctly decoded filename and successfully locate and delete the file.

**Validates: Requirements 2.2, 2.3**

Property 3: Preservation - Non-Upload/Delete API Requests

_For any_ API request that is NOT an upload or delete operation (login, register, chat, transcribe, list manuals, history), the fixed code SHALL produce exactly the same behavior as the original code, preserving Content-Type: application/json headers and JSON request/response handling.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4**

## Fix Implementation

### Changes Required

**File**: `frontend/src/services/manualService.js`

**Function**: `uploadManual`

**Specific Changes**:
1. **Remove Explicit Content-Type Header**: Delete the `headers: { 'Content-Type': 'multipart/form-data' }` line from the `api.post()` call
   - When no Content-Type is explicitly set, the browser automatically sets `Content-Type: multipart/form-data; boundary=<generated-boundary>` for FormData payloads
   - The axios instance's default `Content-Type: application/json` header is only applied when no Content-Type is set by the request config AND the request body is not FormData
   - Axios detects FormData payloads and allows the browser to set the correct Content-Type with boundary

2. **Preserve onUploadProgress Handler**: Keep the `onUploadProgress` callback unchanged to maintain progress tracking functionality

**Function**: `deleteManual`

**Specific Changes**:
3. **Remove encodeURIComponent Call**: Change the URL construction from:
   ```javascript
   `/api/manuals/${encodeURIComponent(filename)}`
   ```
   to:
   ```javascript
   `/api/manuals/${filename}`
   ```
   - Axios automatically handles URL encoding at the HTTP transport level
   - Flask's `<path:filename>` route converter automatically URL-decodes the path segment
   - By removing the pre-encoding, we allow the standard HTTP encoding/decoding flow to work correctly

4. **No Backend Changes Required**: The Flask routes in `backend/flask_api.py` are already correctly implemented with `<path:filename>` route converter and multipart parsing support

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bugs on unfixed code, then verify the fixes work correctly and preserve existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bugs BEFORE implementing the fix. Confirm the root cause analysis by observing the exact failure modes.

**Test Plan**: Write tests that simulate file uploads and manual deletions with various filenames. Run these tests on the UNFIXED code to observe failures and confirm the root causes (missing boundary parameter for uploads, double-encoded filenames for deletes).

**Test Cases**:
1. **Upload with Simple Filename**: Upload `test.pdf` (will fail on unfixed code - no boundary)
2. **Upload with Spaces in Filename**: Upload `Test Manual.pdf` (will fail on unfixed code - no boundary)
3. **Delete with Simple Filename**: Delete `test.pdf` (may succeed on unfixed code if no special chars)
4. **Delete with Spaces in Filename**: Delete `Tesla Model3.pdf` (will fail on unfixed code - double encoding)
5. **Delete with Special Characters**: Delete `Owner's Manual.pdf` (will fail on unfixed code - double encoding)

**Expected Counterexamples**:
- Upload requests fail with 400 Bad Request or multipart parsing errors
- Delete requests for files with spaces/special characters fail with 404 Not Found
- Inspection of request headers shows missing boundary parameter for uploads
- Inspection of Flask logs shows double-encoded filenames for deletes

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds, the fixed functions produce the expected behavior.

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  IF input.operation == 'upload' THEN
    result := uploadManual_fixed(input.file)
    ASSERT result.status == 201
    ASSERT fileExistsOnDisk(input.file.name)
  ELSE IF input.operation == 'delete' THEN
    result := deleteManual_fixed(input.filename)
    ASSERT result.status == 200
    ASSERT NOT fileExistsOnDisk(input.filename)
  END IF
END FOR
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug condition does NOT hold, the fixed code produces the same result as the original code.

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT api_request_original(input) = api_request_fixed(input)
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many test cases automatically across the input domain
- It catches edge cases that manual unit tests might miss
- It provides strong guarantees that behavior is unchanged for all non-upload/delete requests

**Test Plan**: Observe behavior on UNFIXED code first for login, register, chat, transcribe, list manuals, and history requests, then write property-based tests capturing that behavior.

**Test Cases**:
1. **Login Preservation**: Verify POST to `/api/auth/login` continues to send JSON and receive JWT token
2. **Register Preservation**: Verify POST to `/api/auth/register` continues to send JSON
3. **Chat Preservation**: Verify POST to `/api/chat` continues to send JSON message and receive JSON answer
4. **Transcribe Preservation**: Verify POST to `/api/chat/transcribe` continues to send multipart audio (already working)
5. **List Manuals Preservation**: Verify GET to `/api/manuals` continues to receive JSON array
6. **History Preservation**: Verify GET/POST to `/api/history` continues to use JSON
